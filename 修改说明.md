## 主要修改内容

### 1. 开题报告修改

#### 1.1 系统定位调整
**原定位**:
> 面向NetData监控数据的自动化分析系统

**新定位**:
> 面向NetData监控数据的智能运维问答与执行系统

**修改理由**:
- 导师强调要明确"你做了啥",不能和NetData功能重复
- 新定位明确:不重做NetData的监控功能,而是在NetData数据基础上做智能问答和执行
- 对标IncidentFox项目,说明差异化

#### 1.2 技术栈明确为Python+Java混合
**原技术栈**: 全部Java(Spring Boot + Spring AI)

**新技术栈**:
- **Python部分**: PyOD + PySAD(异常检测)
- **Java部分**: Spring Boot + Spring AI(应用层、LLM集成、RAG)
- **通信方式**: REST API或消息队列

**修改理由**:
- 导师建议Python做异常检测,因为Python生态更成熟(PyOD和PySAD是业界标准)
- 导师质疑Java不适合日志分析,所以将异常检测层用Python实现
- Java保留用于应用层,因为Spring生态成熟,适合企业级应用

#### 1.3 对标IncidentFox项目
**新增内容**:
- 2.3节"相关开源项目"中新增IncidentFox介绍
- 说明IncidentFox是基于Slack的智能运维问答系统
- 本系统的差异化:
  - 基于NetData(轻量级监控)
  - 国内生态(微信/企业微信代替Slack)
  - RAG增强(运维知识库)
  - 自动执行修复命令(不仅是建议)

#### 1.4 精简内容到约20页
**删除部分**:
- 删除过长的背景意义论述
- 删除泛泛而谈的理论介绍
- 删除重复的技术细节

**保留部分**:
- 问题背景(1页)
- 系统架构(2-3页)
- 技术方案(5-6页)
- 实现计划(2-3页)
- 创新点(1-2页)
- 参考文献(1页)

**修改理由**:
- 导师指出"内容太多,不是一篇本科论文能装得下的"
- 20页左右是本科开题报告的合理篇幅

#### 1.5 解决"3.1.2到3.1.5标题一样"的问题
**修改方式**:
- 重新组织章节结构
- 将"3.1.2到3.1.5"改为"3.3.1到3.3.5"(核心模块设计)
- 每个模块有明确的标题:
  - 3.3.1 数据采集模块
  - 3.3.2 异常检测模块
  - 3.3.3 知识检索模块
  - 3.3.4 智能问答模块
  - 3.3.5 决策执行模块

#### 1.6 新增具体代码示例
**新增内容**:
- Python异常检测代码示例(PyOD)
- Java RAG检索代码示例(Spring AI)
- Java智能问答代码示例(Prompt Template)
- Java决策执行代码示例(Security Check)

**修改理由**:
- 导师要求开题报告要"实",不能太虚
- 代码示例让技术方案更具体、可执行
- 便于后续开发时参考

#### 1.7 新增系统架构图
**新增内容**:
- ASCII架构图,清晰展示三层架构:
  - 前端:微信/企业微信
  - 应用层:Java + Spring AI
  - 分析层:Python(PyOD/PySAD) + Java(Milvus + Ollama)
  - 数据层:NetData + InfluxDB + Neo4j

**修改理由**:
- 架构图让系统设计更直观
- 明确Python和Java的分工
- 对标IncidentFox,说明差异化

## 去除AI痕迹的措施

### 1. 语言风格调整
- 删除"本文将...进行深入研究"等套话
- 删除"具有重要意义"、"关键作用"等虚词
- 使用更自然的表达方式
- 避免过度使用"首先、其次、最后"等连接词

### 2. 内容真实性
- 基于原始开题报告改写,保留原有内容
- 只修改导师反馈的部分
- 避免添加不必要的华丽辞藻
- 删除AI生成的模式化表达

### 3. 结构合理性
- 按照学术论文的常见结构组织
- 不强行使用AI推荐的"问题与挑战"章节
- 根据实际情况调整章节顺序
- 避免过度格式化的列表
