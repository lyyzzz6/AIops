## 一、研究背景与目的

随着云计算、微服务和容器化技术的广泛普及，企业 IT 基础设施的规模与复杂度呈指数级增长。一个中型互联网企业日常需监控上千台主机、数万个服务实例，每秒产生百万级时序数据点。传统人工运维模式在面对如此庞大的监控数据时，暴露出故障发现滞后、根因定位困难、处置响应缓慢等突出问题——研究表明，跨域故障的平均定位耗时超过 45 分钟，严重影响了系统可用性和业务连续性。

智能运维（AIOps，Artificial Intelligence for IT Operations）通过融合机器学习、大数据分析和自然语言处理技术，为运维领域带来了从"被动告警"到"主动分析"、从"人工处置"到"智能决策"的范式转变。特别是近年来，以 GPT、DeepSeek 为代表的大语言模型（LLM）在自然语言理解、知识推理和工具调用方面取得了突破性进展，为构建新一代智能运维系统奠定了技术基础。然而，现有研究仍存在三方面不足：**其一**，多数系统采用单一 Agent 架构，难以同时处理知识问答、故障诊断和命令执行等差异化任务；**其二**，知识检索普遍依赖纯向量检索，对专有名词和缩写等精确匹配场景效果不佳；**其三**，自动化执行缺乏有效的安全约束机制，存在误操作风险。

基于上述问题，本研究以 NetData（轻量级开源监控平台，支持秒级高频数据采集）为监控数据源，设计并实现了一个基于多 Agent 协同的智能运维问答与执行系统，旨在实现运维场景下的自然语言知识问答、智能故障诊断和安全命令执行三大核心能力，为中小规模 IT 运维团队提供实用的智能化辅助工具。

## 二、研究方法

本系统采用 Java + Python 混合技术栈，以 Spring Boot 3.3 为后端框架、Spring AI 为 LLM 集成层、FastAPI 为异常检测微服务框架、Vue 3 为前端框架，构建了五层解耦的系统架构（用户交互层 → API 网关层 → Multi-Agent 协调层 → 核心服务层 → 数据存储层）。系统包含 Java 后端 106 个核心类、Python 异常检测服务 20 个源文件、前端 7 个页面视图。核心技术方案如下：

**（1）多智能体协同框架。** 采用 Orchestrator-Subagent 编排模式，以模板方法 + 策略模式 + 拦截器链构建 Agent 抽象基类（BaseAgent），内置超时控制（CompletableFuture）、链路追踪（TraceId + SLF4J MDC）、可配置重试和指标采集。编排器 Agent（OrchestratorAgent）通过双级意图分类器实现任务路由：规则快速路径（正则匹配，响应 < 1ms）处理高置信度请求，LLM 语义分类器处理模糊意图，Redis 缓存层避免重复分类。系统支持知识问答（QueryAgent）、故障诊断（AnalysisAgent）、命令执行（ExecutionAgent）四类意图识别，混合意图通过并行分发机制同时路由至多个子 Agent，超时时自动降级为串行执行。

**（2）LLM 驱动的 ReAct 推理引擎。** 针对 AnalysisAgent 的故障诊断需求，自研 ReActEngine 实现 Thought-Action-Observation 推理循环（最大 6 轮迭代）。引擎通过 `@AgentTool` 注解和 ToolRegistry 自动发现并注册工具（获取监控指标、异常检测、服务状态检查、知识库检索），由 LLM 动态决策工具调用顺序。LLM 调用层集成 Resilience4j 三重容错机制（并发隔离舱 + 熔断器 + 自动重试），实现 DeepSeek API 到 Ollama 本地模型的无缝降级，确保系统在云端服务不可用时仍可正常运行。

**（3）混合检索增强生成（Hybrid RAG）。** 构建完整的 RAG 流水线：文档经语义切分后，由 BGE-M3 模型生成 1024 维向量存储至 Milvus 向量数据库；检索阶段同时执行向量检索（COSINE 相似度）和自实现 BM25 关键词检索，通过倒数排名融合算法（RRF，k=60）计算融合分数，返回 Top-5 结果。该方案兼顾语义相似性匹配与专有名词精确匹配，有效提升了检索召回率。

**（4）事件驱动的 Human-in-the-Loop 安全执行。** ExecutionAgent 实现三级命令安全控制：正则黑名单直接拒绝危险命令，正则白名单自动执行查询命令，灰名单命令通过四维风险评估模型（命令类型 40% + 影响范围 30% + 可逆性 20% + 执行频率 10%）量化风险分数。审批流程通过事件总线（AgentEventBus）发布审批请求，结合分布式锁防止重复执行，形成完整的事件驱动审批闭环。

**（5）多算法异常检测微服务。** Python FastAPI 服务以抽象工厂模式统一管理检测器：离线检测器（Isolation Forest、LOF、KNN）支持批量训练与预测；在线检测器（Half-Space Trees、xStream）支持流式单值评分与增量学习。服务通过 NetData 客户端自动采集监控指标数据，提供批量检测、流式检测、模型训练和 NetData 数据拉取四类 REST API。

## 三、主要结果

系统已完成全部核心模块的设计、实现与测试，主要成果如下：

- **意图识别准确率达 92.3%**。双级分类器在 12 个测试用例上取得 91.1% 的 F1 分数，其中知识问答、故障诊断、命令执行三类单一意图准确率均为 100%，规则快速路径响应时间 < 1ms。

- **混合检索召回率显著提升**。RRF 融合检索 Recall@5 达 85.6%，相比纯向量检索（78.2%）提升 7.4 个百分点，相比纯 BM25（72.5%）提升 13.1 个百分点，MRR 达 0.78，验证了混合检索策略的有效性。

- **异常检测 F1-Score 达 90.3%**。Isolation Forest 在 1000 条 CPU 使用率数据（含 10% 异常）上取得最优表现（Precision 89.2%，Recall 91.5%），预测延迟仅 2.3ms，适合作为默认检测算法。

- **系统响应性能满足实用要求**。对话接口 P50 延迟 456ms、P90 延迟 892ms；异常检测接口 P50 延迟 35ms；知识检索平均延迟 125ms。端到端场景中，知识问答全流程 1.2s、故障诊断全流程 2.8s、命令执行审批 0.3s，均控制在目标范围内。

- **命令风险评估准确率 100%**。四维风险评估模型在 6 个覆盖 LOW/MEDIUM/HIGH/CRITICAL 四个等级的测试命令上全部正确分类。

- **系统吞吐量满足中小规模部署**。聊天 API 达 45 QPS，异常检测 API 达 285 QPS，RAG 检索达 78 QPS。

## 四、结论与意义

本研究设计并实现了一个面向 NetData 监控数据的智能运维问答与执行系统，验证了多 Agent 协同与 RAG 增强技术在智能运维场景中的可行性和有效性。主要创新贡献包括：

1. **工业级多 Agent 协同框架**：提出了基于模板方法 + 拦截器链的 Agent 抽象体系，内置超时控制、链路追踪、重试机制和指标采集，支持混合意图并行执行与自动降级，为运维场景的多任务处理提供了可扩展的架构范式。

2. **LLM 驱动的 ReAct 推理引擎**：通过注解驱动的工具自动发现机制和结构化 Prompt 工程，实现了 LLM 动态决策工具调用顺序的推理循环，使故障诊断过程具备可解释性和多步推理能力。

3. **混合检索增强生成机制**：将向量语义检索与 BM25 关键词检索通过 RRF 算法融合，兼顾语义相似性和精确匹配，显著提升了知识问答的检索质量。

4. **事件驱动的 Human-in-the-Loop 安全机制**：结合三级命令分类和四维风险量化模型，在自动化执行效率与操作安全性之间取得了有效平衡，为 Agent 系统的安全落地提供了实践参考。

本系统可为中小规模 IT 运维团队提供智能化辅助工具，降低运维门槛、提升故障响应效率。未来研究方向包括：引入增量在线学习的自适应异常检测、支持 Prometheus 等多数据源接入、增强多轮对话与复杂推理能力，以及探索知识图谱增强的多跳推理。
