## 1 课题背景与意义

### 1.1 研究背景

随着企业IT基础设施越来越复杂,运维人员每天要处理海量监控数据。传统运维模式主要依赖人工经验,这种方式效率低、响应慢。Wang等[1]的研究显示,传统人工运维模式下,跨域故障根因定位平均耗时超过45分钟,即使是简单故障也需要数分钟才能完成处理。

在这种情况下,智能运维(AIOps)成为解决这个问题的方向。通过引入人工智能技术,实现从"被动告警"到"主动分析"、从"人工处置"到"自主决策"的转变。伍文华等[20]指出,随着大语言模型的发展,生成式AI技术在智能运维领域展现出巨大潜力,可以高效利用长期积累的运维经验和技术资料。

NetData是一款轻量级监控工具,它的CPU占用率低于1%,内存消耗仅数十MB,支持1秒级的高频数据采集。这些特性使得NetData可以部署在生产服务器上,不会影响被监控系统的性能,同时提供高保真的监控数据。

另一方面,大语言模型(LLM)在自然语言理解、知识推理、代码生成等方面的能力提升,为构建智能运维Agent提供了技术支持。通过LLM,系统可以理解运维文档、分析日志、生成修复建议。

### 1.2 研究意义

本系统的研发可以直接提升运维效率:

1. 故障检测更实时。通过NetData的高频采集和实时分析,可以在秒级检测到异常,比传统分钟级、小时级的发现时间大幅缩短。

2. 故障诊断更智能。通过分析Agent的智能推理,系统可以自动分析故障根因,生成结构化的诊断报告。Liu等[4]的研究表明,结合LLM和动态知识图谱的方法可以将平均修复时间(MTTR)降低67%。

3. 故障处置更自动化。对于标准化故障,系统可以自动生成并执行修复脚本。对于复杂故障,系统可以提供结构化建议,辅助运维人员决策。牟大恩等[15]的工程实践表明,融合RAG技术的运维助手可以有效提升问答准确性。

4. 运维知识自动沉淀。系统可以自动记录故障处理过程,提取故障模式和处理方案,沉淀为可复用的运维知识。

## 2 国内外研究现状

### 2.1 国外研究现状

#### 2.1.1 智能运维大模型研究

Wang等[1]提出了面向智能运维的集成诊断、预后和决策的大模型。该研究针对工业预测与健康管理(PHM)领域,提出了协作大-小模型范式。专用信号模型提供准确的设备状态评估,大语言模型提供全面的维护指导和决策支持。

#### 2.1.2 多智能体运维框架研究

Niu等[5]针对人工智能数据中心(AIDC)的运维挑战,设计了多智能体运维框架。该框架通过协调器、规划器、工具和裁判等专业智能体的协作,实现了故障诊断的端到端自动化。在包含80个注入故障的基准数据集上,实现了85.16%的根因定位准确率。

#### 2.1.3 RAG增强的多智能体系统研究

Wang等[6]针对电力信息系统异常检测和告警处理,提出了混合RAG增强的LLM多智能体系统。该研究集成了稀疏检索和密集检索,提高了领域知识召回率和事实一致性。

#### 2.1.4 跨域故障定位研究

Ma等[7]针对运营商网络中光传输网络(OTN)和IP无线接入网络(IPRAN)的跨域故障定位,提出了基于大语言模型的智能方法CrossRCA。该方法利用预训练LLM结合提示工程和RAG技术,封装领域特定知识并实现跨异构网络域的复杂上下文推理。

#### 2.1.5 RAG与MCP集成的故障诊断研究

Dai等[9]提出了DiagAgent框架,集成RAG和MCP(Model Context Protocol)工具实现电力设备故障诊断。该框架通过MCP协议标准化地接入外部数据源,结合RAG检索专业知识库,实现端到端的智能诊断。Yang等[10]开发了基于LLM的告警偏差优化Agent,采用ReAct推理范式和RAG技术,实现了设备健康模型告警的自动识别和优化。

#### 2.1.6 知识图谱与Agent决策研究

Yi等[17]提出了LLM-Agent与知识图谱集成框架,用于水电机组维护资源调度的协同优化。该框架通过查询感知的子图检索方法和基于提示工程的三阶段资源分配生成机制,实现了维护知识的动态检索与利用。Zhang等[19]构建了TCN+Transformer多任务诊断模型,结合PPO强化学习算法,在换流器设备运维场景中实现了89.9%的场景识别Macro-F1分数。Hou等[18]针对电网故障报告生成,提出了基于多源异构数据融合和RAG-LLM协作框架的智能生成系统。

## 3 研究开发的基本内容

### 3.1 系统定位

本系统是一个基于NetData监控数据的智能运维问答与执行系统。它不重做NetData的监控功能,而是在NetData采集的数据基础上,通过LLM提供运维建议和自动化执行。

系统采用Orchestrator-Subagent模式,由编排Agent统一接收用户输入或告警事件,识别意图后路由给对应的子Agent处理:

- **编排Agent**: 接收输入,判断意图(知识问答/故障诊断/命令执行),路由给子Agent,汇总结果。
- **问答Agent**: 走RAG流程,检索运维知识库回答问题。
- **分析Agent**: 采用ReAct模式(思考-行动-观察循环),多步工具调用,输出结构化诊断报告。
- **执行Agent**: 生成修复命令,经风险评估和人工审批后执行,记录审计日志。

#### 3.2.1 技术栈选择

- PyOD:用于无监督异常检测
- PySAD:用于流式数据异常检测
- 原因:Python生态在异常检测领域更成熟,这两个库是业界标准

**Java部分(应用层)**:

- Spring AI:与大模型集成,统一抽象接口
- MyBatis-Plus:数据访问
- Milvus:向量数据库,RAG检索
- Neo4j:知识图谱(可选)

**前端**:
- Vue.js 3 + Element Plus:聊天界面和运维工单界面

**LLM选型**:
- DeepSeek-V3 API:主用,用于生产环境

**通信方式**:
- Python和Java之间通过REST API通信
- Python处理异常检测结果,发送给Java层
- Java层调用LLM,生成运维建议

### 3.3 核心模块设计

#### 3.3.1 数据采集模块

- 通过NetData的REST API和WebSocket接口采集监控指标
- 采集频率为1秒
- 采集内容包括CPU、内存、磁盘、网络等多维度指标
- 监控数据通过NetData API实时获取,异常检测结果存储到MySQL

#### 3.3.2 异常检测模块

- 使用PyOD进行无监督异常检测
- 使用PySAD进行流式数据异常检测
- 检测到异常后,将结果通过REST API发送给Java层
- 异常数据包括:异常时间、异常指标、异常值、历史基线等

**代码示例(Python)**:

```python
from pyod.models.iforest import IForest
import numpy as np

# 加载监控数据
data = np.load('netdata_metrics.npy')

# 训练Isolation Forest模型
clf = IForest(contamination=0.01)
clf.fit(data)

# 检测异常
anomalies = clf.predict(data)

# 将异常结果发送给Java层
requests.post('http://localhost:8080/api/anomaly', 
              json={'anomalies': anomalies.tolist()})
```

#### 3.3.3 知识检索模块

- 构建运维知识库,包含历史故障案例、处理方案、最佳实践
- 使用Milvus向量数据库进行向量检索
- 使用BM25进行关键词检索
- 通过RRF(Reciprocal Rank Fusion)融合两路检索结果,再经bge-reranker精排,返回最相关的知识片段

**代码示例(Java)**:

```java
@Service
public class KnowledgeRetrievalService {
    
    @Autowired
    private MilvusService milvusService;
    
    @Autowired
    private SearchService bm25Service;
    
    public List<String> retrieve(String query) {
        // 向量检索
        List<String> vectorResults = milvusService.search(query);
        
        // 关键词检索
        List<String> keywordResults = bm25Service.search(query);
        
        // 混合检索
        return combineResults(vectorResults, keywordResults);
    }
}
```

#### 3.3.4 智能问答模块

- 使用Spring AI与DeepSeek API集成
- 使用Prompt Template管理提示词模板
- 将异常数据、检索到的知识片段作为上下文
- 生成自然语言的运维建议

**代码示例(Java)**:

```java
@Service
public class ChatService {
    
    @Autowired
    private ChatClient chatClient;
    
    public String generateSuggestion(AnomalyData anomaly, 
                                      List<String> knowledge) {
        String prompt = """
            你是一个运维专家。
            
            当前异常情况:
            - 异常时间: %s
            - 异常指标: %s
            - 异常值: %s
            - 历史基线: %s
            
            相关知识:
            %s
            
            请生成运维建议,包括:
            1. 可能的原因
            2. 建议的解决方案
            3. 需要执行的命令(如果有的话)
            """.formatted(anomaly.getTime(), anomaly.getMetric(), 
                         anomaly.getValue(), anomaly.getBaseline(),
                         String.join("\n", knowledge));
        
        return chatClient.prompt(prompt)
                         .call()
                         .content();
    }
}
```

#### 3.3.5 决策执行模块

- 将运维建议转化为可执行的命令
- 支持Linux命令、SQL脚本、配置修改等
- 提供安全审核机制,基于命令白名单和风险评估矩阵
- 高风险操作需要人工确认(Human-in-the-Loop)

**代码示例(Java)**:

```java
@Service
public class ExecutionService {
    
    @Autowired
    private SecurityCheckService securityCheck;
    
    public ExecutionResult executeCommand(String command, 
                                          String user) {
        // 安全审核
        RiskLevel risk = securityCheck.check(command);
        
        if (risk == RiskLevel.HIGH) {
            // 高风险操作,需要人工确认
            return ExecutionResult.pendingApproval();
        }
        
        // 执行命令
        Process process = Runtime.getRuntime().exec(command);
        
        // 返回执行结果
        return ExecutionResult.success(process.getInputStream());
    }
}
```

## 4 创新点

### 4.1 针对NetData的专用智能Agent设计

现有研究多针对通用监控数据或特定行业的专用数据,缺乏针对NetData这一特定监控平台的智能Agent设计。本系统针对NetData的高频采集(1秒级)、超低资源占用(CPU<1%)等特性,设计了专门的数据适配和异常感知流程。

### 4.2 Orchestrator-Subagent多Agent协同架构

系统采用编排Agent统一调度问答、分析、执行三个子Agent的架构,通过意图识别实现任务的智能路由。相比单Agent方案,多Agent协同可以让每个Agent专注于特定任务,提升整体系统的准确性和可维护性。

### 4.3 自动化执行与Human-in-the-Loop

IncidentFox只提供建议,本系统不仅可以提供建议,还能自动执行修复命令(在安全范围内)。通过风险评估矩阵和命令白名单机制,高风险操作需要人工确认,兼顾效率和安全。

### 4.4 混合检索RAG增强

构建运维知识库,采用向量检索+关键词检索的混合检索方案,经RRF融合和Reranker精排后注入LLM上下文,缓解大模型"幻觉"问题,提升回答准确性。
